The slow query table is the core view in DBSight. It surfaces the queries consuming the most time on your database, ranked by total execution cost.

## How It Works

DBSight reads from database-specific statistics tables:

- **PostgreSQL:** `pg_stat_statements` extension
- **MySQL/MariaDB:** `performance_schema.events_statements_summary_by_digest`

These tables accumulate cumulative statistics — total calls, total time, rows returned — for every normalized query fingerprint your database has seen since the last reset.

The background worker collects a snapshot on each polling interval. Between snapshots, DBSight computes **deltas** so you can see which queries have gotten worse since the last collection pass, not just which queries are worst overall.

## Database-Specific Setup

### PostgreSQL: pg_stat_statements

The monitored database must have `pg_stat_statements` installed and loaded:

```sql
-- Check if it is already active
SELECT * FROM pg_stat_statements LIMIT 1;

-- If not, install it (requires superuser)
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
```

Add it to `shared_preload_libraries` in `postgresql.conf` and restart PostgreSQL if needed.

### MySQL 5.7+ / 8.0+: performance_schema

Ensure `performance_schema` is enabled (usually on by default):

```sql
-- Check if it is enabled
SHOW GLOBAL VARIABLES LIKE 'performance_schema';
-- Should return 'ON'

-- If disabled, add to my.cnf and restart:
[mysqld]
performance_schema = ON

-- Timer conversion: MySQL stores times in picoseconds
-- DBSight converts: picoseconds / 1e9 = milliseconds
```

### MariaDB 10.x+: performance_schema

Enable `performance_schema` in the configuration:

```sql
-- Check if it is enabled
SHOW GLOBAL VARIABLES LIKE 'performance_schema';
-- Should return 'ON'

-- If disabled, add to my.cnf and restart:
[mysqld]
performance_schema = ON

-- Note: MariaDB lacks some fields (QUANTILE_95, FIRST_SEEN, LAST_SEEN)
-- but basic metrics work the same way as MySQL
```

## Query Table Columns

| Column          | Description                                                  |
| --------------- | ------------------------------------------------------------ |
| **Query**       | Normalized query text (literals replaced with `$1`, `$2`, …) |
| **Calls**       | Total number of times this query has been executed           |
| **Total Time**  | Cumulative execution time in milliseconds                    |
| **Mean Time**   | Average execution time per call                              |
| **Rows**        | Total rows returned or affected                              |
| **Delta Calls** | New calls since the previous snapshot                        |
| **Delta Time**  | Additional total time since the previous snapshot            |

## Sorting

Click any column header to sort by that metric. Sorting by **Total Time** (default) identifies the queries with the highest cumulative cost. Sorting by **Mean Time** finds individually slow queries that may be called infrequently. Sorting by **Delta Time** highlights what is actively getting worse right now.

## Delta Tracking

Each time the worker collects a snapshot, it stores the cumulative counters. The delta columns show the difference between the two most recent snapshots for each query fingerprint. A large delta in a short interval indicates a query that is suddenly running more often or taking longer — worth investigating immediately.

## Sparkline Charts

Each query row includes a small sparkline chart visualizing total time across the last several snapshots. A flat line means stable performance. An upward trend means the query is accumulating cost over time.

## Auto-Refresh via SSE

The query table subscribes to a Server-Sent Events stream at `/api/connections/:id/queries/stream`. New data pushes to the browser automatically each time the worker completes a collection pass — no manual refresh needed. See [Live Dashboard](/en/user-guide/06-live-dashboard) for details on the SSE mechanism.
