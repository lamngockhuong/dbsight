## PostgreSQL Database Statistics

Source: `internal/adapter/stats.go` — `GetDatabaseStats`

```sql
SELECT
    current_database(),
    pg_database_size(current_database()),
    (SELECT count(*) FROM pg_stat_activity WHERE state = 'active'),
    current_setting('max_connections')::int,
    ROUND(
        sum(blks_hit)::numeric /
        NULLIF(sum(blks_hit) + sum(blks_read), 0) * 100, 2
    )
FROM pg_stat_database WHERE datname = current_database()
```

The query selects from `pg_stat_database` filtered to the current database, combining it with a correlated subquery for active connections and a `current_setting` call for the configured connection limit.

### Metrics Reference

#### Database Name

```sql
current_database()
```

Returns the name of the database the connection is authenticated against. Used as the display label in the dashboard header.

#### Database Size

```sql
pg_database_size(current_database())
```

Returns the total on-disk size of the database in bytes, including all tables, indexes, and TOAST data. DBSight formats this into human-readable units (KB / MB / GB) for display.

#### Active Connections

```sql
(SELECT count(*) FROM pg_stat_activity WHERE state = 'active')
```

Counts connections currently executing a query (`state = 'active'`). This excludes idle connections, idle-in-transaction connections, and background workers.

| `pg_stat_activity.state` | Meaning                                       |
| ------------------------ | --------------------------------------------- |
| `active`                 | Executing a query right now — counted here    |
| `idle`                   | Connected but waiting for a command           |
| `idle in transaction`    | Inside a BEGIN block, not currently executing |
| `fastpath function call` | Executing a fast-path function                |

#### Max Connections

```sql
current_setting('max_connections')::int
```

The server-wide connection limit from `postgresql.conf`. DBSight displays active connections alongside this value so you can see headroom at a glance.

#### Cache Hit Ratio

```sql
ROUND(
    sum(blks_hit)::numeric /
    NULLIF(sum(blks_hit) + sum(blks_read), 0) * 100, 2
)
```

The percentage of block reads served from PostgreSQL's shared buffer cache rather than from disk. `NULLIF` guards against division-by-zero on a fresh database with no I/O yet.

| Range     | Status   | Interpretation                                                              |
| --------- | -------- | --------------------------------------------------------------------------- |
| >= 99%    | Good     | Nearly all reads from memory. Expected for a well-tuned, warm database.     |
| 95% – 99% | Warning  | Some disk reads occurring. May be acceptable depending on dataset size.     |
| < 95%     | Critical | Significant disk I/O. Consider increasing `shared_buffers` or adding RAM.   |
| `NULL`    | N/A      | No block I/O recorded yet — database just started or statistics were reset. |

A low cache hit ratio does not always mean misconfiguration. Databases larger than available RAM will always have some disk reads; the key is whether the ratio is declining over time.

## MySQL Database Statistics

MySQL does not provide a single system view for all database-level metrics. DBSight issues multiple queries to `SHOW GLOBAL STATUS`, `SHOW VARIABLES`, and `information_schema.TABLES`.

Source: `internal/adapter/mysql_stats.go` — `GetDatabaseStats`

### Database Name

```sql
SELECT DATABASE()
```

Returns the name of the currently selected database.

### Database Size

```sql
SELECT IFNULL(SUM(data_length + index_length), 0)
FROM information_schema.TABLES
WHERE table_schema = DATABASE()
```

Sums `data_length` (row data) and `index_length` (index data) for all tables in the current database. This is an estimate based on InnoDB statistics; run `ANALYZE TABLE` on key tables for more accurate values.

### Active Connections

```sql
SHOW GLOBAL STATUS LIKE 'Threads_connected'
```

Returns the current number of open client connections. Unlike PostgreSQL's `pg_stat_activity`, this counts all connected threads regardless of their current state (active, idle, waiting).

### Max Connections

```sql
SHOW VARIABLES LIKE 'max_connections'
```

The server-wide connection limit from `my.cnf` or set at runtime with `SET GLOBAL max_connections`.

### InnoDB Buffer Pool Cache Hit Ratio

DBSight computes the cache hit ratio from two status variables:

```sql
SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_reads'
SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_read_requests'
```

The ratio is calculated as:

```
cache_hit_ratio = (1 - Innodb_buffer_pool_reads / Innodb_buffer_pool_read_requests) * 100
```

| Variable                           | Description                                       |
| ---------------------------------- | ------------------------------------------------- |
| `Innodb_buffer_pool_reads`         | Physical disk reads (cache misses)                |
| `Innodb_buffer_pool_read_requests` | Total logical read requests (cache hits + misses) |

These are cumulative counters since server start. A high ratio (> 99%) indicates most reads are served from the InnoDB buffer pool.

| Range     | Status   | Interpretation                                                           |
| --------- | -------- | ------------------------------------------------------------------------ |
| >= 99%    | Good     | Nearly all reads from buffer pool memory.                                |
| 95% – 99% | Warning  | Some disk reads. May be acceptable depending on dataset size.            |
| < 95%     | Critical | Significant disk I/O. Consider increasing `innodb_buffer_pool_size`.     |
| `0`       | N/A      | No read requests yet — server just started or no InnoDB tables accessed. |

## MariaDB Database Statistics

MariaDB uses the same query approach as MySQL. Both engines share the `SHOW GLOBAL STATUS`, `SHOW VARIABLES`, and `information_schema.TABLES` interfaces.

Source: `internal/adapter/mariadb_stats.go` — `GetDatabaseStats`

### Database Name

```sql
SELECT DATABASE()
```

### Database Size

```sql
SELECT IFNULL(SUM(data_length + index_length), 0)
FROM information_schema.TABLES
WHERE table_schema = DATABASE()
```

### Active Connections

```sql
SHOW GLOBAL STATUS LIKE 'Threads_connected'
```

### Max Connections

```sql
SHOW VARIABLES LIKE 'max_connections'
```

### InnoDB Buffer Pool Cache Hit Ratio

```sql
SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_reads'
SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_read_requests'
```

Calculated identically to MySQL:

```
cache_hit_ratio = (1 - Innodb_buffer_pool_reads / Innodb_buffer_pool_read_requests) * 100
```

### MariaDB vs MySQL Differences

The database statistics queries are functionally identical between MySQL and MariaDB. Both engines expose the same `SHOW GLOBAL STATUS` variables and `information_schema.TABLES` schema. There are no behavioral differences for this set of metrics.

One operational note: MariaDB may report `Threads_connected` slightly differently than MySQL when thread pooling is enabled (`thread_handling = pool-of-threads`). In that configuration, `Threads_connected` reflects logical connections rather than OS threads.

## How DBSight Displays These Metrics

The four values map directly to the summary cards on the Live Dashboard:

- **Database** — name label in the page header
- **Size** — formatted bytes card (e.g. `4.2 GB`)
- **Connections** — `{active} / {max_connections}` with a usage bar
- **Cache Hit** — percentage with color coding (green / amber / red per the ranges above)

All four refresh on every SSE tick alongside the slow query stream, so the dashboard reflects the live state of the database without manual page reloads.
