## PostgreSQL EXPLAIN

`EXPLAIN` shows PostgreSQL's query execution plan — the sequence of operations the planner chose to satisfy a query. `EXPLAIN ANALYZE` actually executes the query and adds real timing and row counts alongside the estimates.

### Safety Mechanism

Running arbitrary `EXPLAIN ANALYZE` statements carries risk: a malicious or accidental write query could modify data. DBSight mitigates this by wrapping every EXPLAIN in a read-only transaction that is always rolled back.

Source: `internal/adapter/explain.go`

```sql
BEGIN;
SET TRANSACTION READ ONLY;
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) <query>;
ROLLBACK;
```

The `SET TRANSACTION READ ONLY` call causes PostgreSQL to reject any data-modifying operation within the transaction. The explicit `ROLLBACK` (via `defer tx.Rollback`) ensures no changes persist even if `READ ONLY` were somehow bypassed.

### Two Execution Modes

#### Estimate-only mode (no ANALYZE)

Used when `AnalyzeMode = false`. Returns the planner's cost estimates without executing the query. Safe for any query type and has zero runtime cost.

```sql
EXPLAIN (FORMAT JSON) <query>
```

Use this mode when:

- The query is expensive and you only need the plan structure
- The query has write side-effects you want to avoid executing
- You want instant results without waiting for execution

#### Full analyze mode

Used when `AnalyzeMode = true`. Executes the query inside the read-only transaction and returns actual row counts, loop counts, and per-node timing.

```sql
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) <query>
```

The `BUFFERS` option adds shared/local block hit and read counts, which reveals whether nodes are reading from cache or disk.

### JSON Output Structure

DBSight requests `FORMAT JSON` for all plans. The top-level structure is an array with one element:

```json
[
  {
    "Plan": {
      "Node Type": "Hash Join",
      "Startup Cost": 0.00,
      "Total Cost": 1234.56,
      "Plan Rows": 500,
      "Actual Rows": 487,
      "Actual Loops": 1,
      "Plans": [ "..." ]
    }
  }
]
```

Each node in `Plans` follows the same structure recursively.

### Common Plan Node Types

| Node Type          | Description                                                                                                                       |
| ------------------ | --------------------------------------------------------------------------------------------------------------------------------- |
| `Seq Scan`         | Full table scan — reads every row. Normal for small tables or large result sets; costly on large tables without a `WHERE` clause. |
| `Index Scan`       | Uses an index to locate rows, then fetches from the heap. Good for selective queries.                                             |
| `Index Only Scan`  | Satisfies the query entirely from the index, no heap access. Most efficient read path.                                            |
| `Bitmap Heap Scan` | Uses a bitmap from one or more indexes, then fetches heap pages in order. Good for moderate selectivity.                          |
| `Hash Join`        | Builds a hash table from the smaller relation then probes with the larger. Common for equi-joins on unsorted data.                |
| `Nested Loop`      | For each row in the outer set, scans the inner set. Efficient when inner side uses an index.                                      |
| `Merge Join`       | Joins two pre-sorted inputs. Efficient when both sides are already ordered.                                                       |
| `Sort`             | Explicit sort step. Look for high `Sort Method: external merge` which indicates disk spill.                                       |
| `Aggregate`        | Computes aggregate functions (COUNT, SUM, etc.).                                                                                  |
| `Limit`            | Truncates output to N rows.                                                                                                       |

### What to Look For

- **High `Rows Removed by Filter`** on a `Seq Scan` — an index on the filter column would help.
- **Estimated rows vs actual rows diverging by 10x or more** — stale statistics; run `ANALYZE` on the table.
- **`Sort Method: external merge`** — sort spilled to disk; increase `work_mem`.
- **`Shared Read Blocks` much higher than `Shared Hit Blocks`** — data not in cache; normal for cold queries, investigate if persistent.
- **Nested Loop with many loops and no index on inner side** — consider adding an index or rewriting as a Hash Join.

## MySQL EXPLAIN

MySQL supports `EXPLAIN FORMAT=JSON` for plan estimates and `EXPLAIN ANALYZE` for actual execution stats (MySQL 8.0.18+ only). DBSight runs all EXPLAIN calls inside a read-only transaction.

Source: `internal/adapter/mysql_explain.go`

### Safety Mechanism

```go
tx, err := a.db.BeginTx(ctx, &sql.TxOptions{ReadOnly: true})
defer tx.Rollback()
```

DBSight uses Go's `database/sql` read-only transaction option, which maps to `START TRANSACTION READ ONLY` in MySQL. The transaction is always rolled back after the EXPLAIN call.

### Two Execution Modes

#### Estimate-only mode

Used when `AnalyzeMode = false`. Works on all MySQL 5.7+ and 8.0+ connections.

```sql
EXPLAIN FORMAT=JSON <query>
```

Returns the optimizer's cost estimates as a JSON document without executing the query.

#### Full analyze mode

Used when `AnalyzeMode = true`. Requires **MySQL 8.0.18+** and a `SELECT` query. DBSight rejects non-SELECT queries in this mode to avoid accidental writes.

```sql
EXPLAIN ANALYZE <query>
```

MySQL's `EXPLAIN ANALYZE` returns output in TREE text format (not JSON). DBSight wraps the text in a JSON envelope for consistent handling:

```json
{ "format": "tree", "text": "-> Nested loop inner join  (cost=...)" }
```

### JSON Output Structure (FORMAT=JSON)

```json
{
  "query_block": {
    "select_id": 1,
    "cost_info": {
      "query_cost": "1.05"
    },
    "table": {
      "table_name": "orders",
      "access_type": "ref",
      "key": "idx_user_id",
      "rows_examined_per_scan": 3,
      "rows_produced_per_join": 3,
      "cost_info": {
        "read_cost": "0.75",
        "eval_cost": "0.30",
        "prefix_cost": "1.05"
      }
    }
  }
}
```

### Key Fields

| Field                    | Description                                             |
| ------------------------ | ------------------------------------------------------- |
| `access_type`            | How MySQL accesses the table (`ALL`, `ref`, `range`, …) |
| `key`                    | Index used (if any)                                     |
| `rows_examined_per_scan` | Estimated rows examined per scan                        |
| `cost_info.query_cost`   | Optimizer's estimated total cost                        |

### What to Look For

- **`access_type: ALL`** — full table scan; consider adding an index on the filter/join column.
- **`key: null`** — no index used; review `WHERE` and `JOIN` predicates.
- **High `rows_examined_per_scan` relative to rows returned** — poor selectivity; a more specific index may help.

## MariaDB EXPLAIN

MariaDB supports `EXPLAIN FORMAT=JSON` for plan estimates and `ANALYZE FORMAT=JSON` for actual execution stats (MariaDB 10.1+). The key difference from MySQL is that MariaDB's analyze mode returns JSON rather than TREE text.

Source: `internal/adapter/mariadb_explain.go`

### Safety Mechanism

```go
tx, err := a.db.BeginTx(ctx, &sql.TxOptions{ReadOnly: true})
defer tx.Rollback()
```

Same read-only transaction approach as MySQL. The transaction is always rolled back.

### Two Execution Modes

#### Estimate-only mode

Used when `AnalyzeMode = false`.

```sql
EXPLAIN FORMAT=JSON <query>
```

#### Full analyze mode

Used when `AnalyzeMode = true`. Requires a `SELECT` query. MariaDB uses `ANALYZE FORMAT=JSON` (not `EXPLAIN ANALYZE` like MySQL) and returns JSON directly — no TREE text wrapping needed.

```sql
ANALYZE FORMAT=JSON <query>
```

### JSON Output Structure

MariaDB's `EXPLAIN FORMAT=JSON` output follows a similar structure to MySQL but with some differences:

```json
{
  "query_block": {
    "select_id": 1,
    "table": {
      "table_name": "orders",
      "access_type": "ref",
      "key": "idx_user_id",
      "key_length": "4",
      "used_key_parts": ["user_id"],
      "rows": 3,
      "r_rows": 2.5,
      "r_filtered": 100,
      "filtered": 100
    }
  }
}
```

### MariaDB vs MySQL Differences

| Aspect               | MySQL                                 | MariaDB                                |
| -------------------- | ------------------------------------- | -------------------------------------- |
| Analyze command      | `EXPLAIN ANALYZE` (returns TREE text) | `ANALYZE FORMAT=JSON` (returns JSON)   |
| Analyze availability | MySQL 8.0.18+                         | MariaDB 10.1+                          |
| Actual row field     | Not in FORMAT=JSON output             | `r_rows` in ANALYZE FORMAT=JSON output |
| Actual filter field  | Not in FORMAT=JSON output             | `r_filtered` in ANALYZE FORMAT=JSON    |

### What to Look For

- **`access_type: ALL`** — full table scan; add an index.
- **`r_rows` much higher than `rows`** — actual execution differs significantly from estimates; run `ANALYZE TABLE` to refresh statistics.
- **`r_filtered` far below `filtered`** — optimizer's row estimate is optimistic; index may not be selective enough.
