---
title: Index & Table Statistics
description: SQL queries for index usage, duplicate detection, and table statistics
---

DBSight runs three queries against PostgreSQL system catalogs to surface index health: index usage stats, duplicate index detection, and table-level scan statistics. All three are read-only and safe to run in production.

## Index Usage

Source: `internal/adapter/indexes.go` — `GetIndexStats`

```sql
SELECT
    schemaname,
    relname,
    indexrelname,
    idx_scan,
    idx_tup_read,
    idx_tup_fetch,
    pg_relation_size(indexrelid) AS index_size_bytes
FROM pg_stat_user_indexes
ORDER BY idx_scan ASC, pg_relation_size(indexrelid) DESC
```

Results are sorted by `idx_scan ASC` so unused indexes (0 scans) appear first, then by size descending so large unused indexes are most visible.

### Column Descriptions

| Column             | Type   | Description                                                        |
| ------------------ | ------ | ------------------------------------------------------------------ |
| `schemaname`       | name   | Schema containing the table                                        |
| `relname`          | name   | Table name                                                         |
| `indexrelname`     | name   | Index name                                                         |
| `idx_scan`         | bigint | Number of times the index was used by the planner since last reset |
| `idx_tup_read`     | bigint | Tuples returned from index scans                                   |
| `idx_tup_fetch`    | bigint | Tuples fetched from the heap via this index                        |
| `index_size_bytes` | bigint | On-disk size of the index in bytes                                 |

### Interpreting Results

- **`idx_scan = 0`** — the index has never been used since statistics were last reset. If the database has been running under normal load for days or weeks, this index is a drop candidate. Verify with `SELECT pg_stat_reset()` timing before dropping.
- **Large size with low scans** — the index costs write overhead and disk space with little read benefit.
- **`idx_tup_fetch` much lower than `idx_tup_read`** — the index is used for scans but many fetched tuples are filtered out by visibility or additional predicates, which may indicate a partially matching index.

## Duplicate Indexes

Source: `internal/adapter/indexes.go` — `GetDuplicateIndexes`

```sql
SELECT
    a.tablename,
    a.indexname AS index1,
    b.indexname AS index2,
    a.indexdef
FROM pg_indexes a
JOIN pg_indexes b
    ON a.tablename = b.tablename
    AND a.indexdef = b.indexdef
    AND a.indexname < b.indexname
WHERE a.schemaname NOT IN ('pg_catalog', 'information_schema')
ORDER BY a.tablename, a.indexname
```

The self-join on `pg_indexes` matches pairs where both `tablename` and `indexdef` are identical. The `a.indexname < b.indexname` condition ensures each duplicate pair appears exactly once.

### Column Descriptions

| Column      | Type | Description                                |
| ----------- | ---- | ------------------------------------------ |
| `tablename` | name | Table both indexes belong to               |
| `index1`    | name | First index name (alphabetically earlier)  |
| `index2`    | name | Second index name (alphabetically later)   |
| `indexdef`  | text | Full index definition (identical for both) |

### Interpreting Results

Any row returned here represents two indexes with exactly the same definition. One of them is redundant — it consumes disk space and slows down every `INSERT`, `UPDATE`, and `DELETE` on the table without adding any query capability. Drop one from each pair after confirming neither is used by a unique constraint or foreign key.

Duplicates commonly appear after migrations that recreate indexes without dropping the originals, or after `pg_restore` runs against a database that already has indexes.

## Table Statistics

Source: `internal/adapter/indexes.go` — `GetTableStats`

```sql
SELECT
    schemaname,
    relname,
    seq_scan,
    seq_tup_read,
    n_live_tup,
    pg_total_relation_size(relid) AS table_size_bytes
FROM pg_stat_user_tables
ORDER BY seq_scan DESC
```

Results are sorted by `seq_scan DESC` to surface tables receiving the most full-table scans.

### Column Descriptions

| Column             | Type   | Description                                            |
| ------------------ | ------ | ------------------------------------------------------ |
| `schemaname`       | name   | Schema containing the table                            |
| `relname`          | name   | Table name                                             |
| `seq_scan`         | bigint | Number of sequential (full) scans on this table        |
| `seq_tup_read`     | bigint | Total tuples read via sequential scans                 |
| `n_live_tup`       | bigint | Estimated number of live rows                          |
| `table_size_bytes` | bigint | Total on-disk size including indexes and TOAST (bytes) |

### Interpreting Results

- **High `seq_scan` on a large table** — the most actionable signal. If `seq_tup_read / seq_scan` is close to `n_live_tup`, nearly every scan reads the entire table. An index on the most common filter column will convert these to `Index Scan` operations.
- **High `seq_scan` on a small table** — often acceptable. PostgreSQL's planner intentionally chooses sequential scans on small tables because they are faster than index lookups when most rows qualify.
- **`n_live_tup` much lower than expected** — may indicate aggressive `DELETE` without `VACUUM`, causing table bloat. Check with `pg_relation_size` vs `pg_total_relation_size`.
