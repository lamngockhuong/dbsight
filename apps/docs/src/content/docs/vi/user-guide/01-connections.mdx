Một kết nối cho DBSight biết cơ sở dữ liệu nào cần theo dõi. Bạn có thể thêm bao nhiêu kết nối tùy ý — mỗi kết nối chạy độc lập thông qua Worker nền. DBSight hỗ trợ PostgreSQL, MySQL và MariaDB.

## Thêm Kết nối

1. Nhấp **Add Connection** trong thanh bên trái.
2. Nhập **Tên** — đây chỉ là nhãn hiển thị (ví dụ: `prod-replica`, `staging-db`).
3. Chọn **Loại Cơ sở dữ liệu** từ dropdown (PostgreSQL, MySQL hoặc MariaDB).
4. Nhập **DSN** (Data Source Name) cho cơ sở dữ liệu đích.
   - Bạn có thể chuyển đổi giữa chế độ **DSN** (dán chuỗi kết nối) và chế độ **Form** (điền từng trường).
5. Nhấp **Test Connection** để xác minh kết nối trước khi lưu.
6. Nhấp **Save**.

Kết nối xuất hiện trong thanh bên ngay lập tức. Worker sẽ lập lịch lần thu thập metrics đầu tiên trong vòng một chu kỳ polling.

## Định dạng DSN

### PostgreSQL

Định dạng chuỗi kết nối PostgreSQL tiêu chuẩn:

```
postgresql://user:password@host:port/dbname?sslmode=disable
```

**Ví dụ:**

```bash
# Cơ sở dữ liệu cục bộ, không có SSL
postgresql://postgres:postgres@localhost:5432/mydb?sslmode=disable

# Cơ sở dữ liệu từ xa với SSL bắt buộc
postgresql://appuser:secret@db.example.com:5432/production?sslmode=require

# Từ xa với đường dẫn tìm kiếm schema cụ thể
postgresql://appuser:secret@db.example.com:5432/production?sslmode=require&search_path=myschema
```

Sử dụng người dùng cơ sở dữ liệu chỉ đọc khi có thể. DBSight chỉ cần quyền `SELECT` trên `pg_stat_statements`, `pg_stat_user_indexes`, `pg_stat_user_tables` và `pg_database`.

### MySQL 5.7+ / 8.0+

MySQL sử dụng định dạng DSN khác với driver `go-sql-driver/mysql`:

```
user:password@tcp(host:port)/dbname?parseTime=true&timeout=10s
```

**Ví dụ:**

```bash
# Cơ sở dữ liệu cục bộ
root:password@tcp(localhost:3306)/mydb?parseTime=true&timeout=10s

# Cơ sở dữ liệu từ xa với các tham số cụ thể
appuser:secret@tcp(db.example.com:3306)/production?parseTime=true&timeout=10s&charset=utf8mb4
```

**Yêu cầu:**

- `performance_schema` phải được bật trên MySQL server
- Sử dụng người dùng chỉ đọc với quyền `SELECT` trên `performance_schema.*`, `information_schema.*` và `SHOW GLOBAL STATUS`

### MariaDB 10.x+

MariaDB sử dụng cùng định dạng DSN như MySQL:

```
user:password@tcp(host:port)/dbname?parseTime=true&timeout=10s
```

**Ví dụ:**

```bash
# Cơ sở dữ liệu cục bộ
root:password@tcp(localhost:3306)/mydb?parseTime=true&timeout=10s

# Cơ sở dữ liệu từ xa
appuser:secret@tcp(db.example.com:3306)/production?parseTime=true&timeout=10s
```

**Yêu cầu:**

- `performance_schema` phải được bật trên MariaDB server
- Sử dụng người dùng chỉ đọc với quyền `SELECT` trên `performance_schema.*`, `information_schema.*` và `SHOW GLOBAL STATUS`

## Kiểm tra Kết nối

Nút **Test Connection** gửi một ping nhẹ đến cơ sở dữ liệu đích bằng DSN được cung cấp. Nó kiểm tra:

- Khả năng tiếp cận mạng
- Thông tin xác thực hợp lệ
- `pg_stat_statements` có thể truy cập được

Nếu kiểm tra thất bại, một thông báo lỗi dạng văn bản thuần sẽ được hiển thị. Nguyên nhân thường gặp: sai host/port, thông tin xác thực không đúng, tường lửa chặn kết nối, hoặc `pg_stat_statements` chưa được cài đặt trên máy đích.

## Chỉnh sửa Kết nối

1. Nhấp vào tên kết nối trong thanh bên.
2. Nhấp biểu tượng **Edit** (bút chì) ở phần header của kết nối.
3. Cập nhật tên hoặc DSN.
4. Kiểm tra và lưu.

Chỉnh sửa kết nối không xóa lịch sử đã thu thập.

## Xóa Kết nối

1. Mở kết nối.
2. Nhấp biểu tượng **Delete** (thùng rác).
3. Xác nhận việc xóa.

Xóa một kết nối sẽ xóa nó và tất cả các snapshot liên quan khỏi cơ sở dữ liệu metadata. Hành động này không thể hoàn tác.

## Bảo mật

Các DSN được mã hóa bằng **AES-256-GCM** trước khi lưu vào cơ sở dữ liệu metadata. DSN dạng văn bản thuần không bao giờ được ghi ra đĩa hoặc trả về trong bất kỳ phản hồi API nào — trường `EncryptedDSN` mang tag `json:"-"` đảm bảo nó luôn bị bỏ qua khi serialize.
